# `--del` Flag Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a `--del` one-shot flag to all four runners that deletes this agent's isolated config (`~/.docker-agent/<proj>/<agent>`) for the resolved work dir, after the user types the project name to confirm.

**Architecture:** Mirror the existing `--edit` one-shot: parse `--del` in the long-flag loop, then — after `WORK_DIR`/`ISOLATE` are resolved but before the seeding/`mkdir` block — refuse non-isolated configs, confirm by typed project name, `rm -rf` the agent dir (+ claude's `.json`), prune the empty parent, and exit.

**Tech Stack:** bash launchers.

**Spec:** `docs/superpowers/specs/2026-06-07-del-flag-design.md`

**Branch:** `feat/del-flag` (already checked out; spec commit `e25d065` lives here).

**Note on "tests":** launcher work — verification = `bash -n` plus runtime checks against a throwaway `~/.docker-agent` entry (confirm match deletes, mismatch aborts, `-H`/`-c` refuse, empty-parent prune, gitconfig survives).

---

## File Structure

- `run-claude.sh` — case-style long-flag loop; has `--mem-*`; config var `CONFIG_DIR`; agent `claude`; has sibling `.json`.
- `run-goose.sh` / `run-hermes.sh` / `run-pi.sh` — single-line long-flag style; config var `CONFIG_SRC`; agents `goose`/`hermes`/`pi`; no sibling `.json`.
- `README.md` — flags/options listing.

The deletion block is the same logic in every runner; only the agent name and the
config-var name differ. The `[ -e "$DEL_DIR.json" ]` line is uniform (no-op where there
is no sibling).

```sh
# Resolve the work dir up front; -i derives the config dir name from it.
WORK_DIR="$(cd "$WORK_DIR" && pwd)"

# --del: remove this agent's isolated config for the work dir, then exit. Only
# touches the isolated ~/.docker-agent path — never -H host config or -c custom.
if [ "$DEL" -eq 1 ]; then
  if [ "$ISOLATE" -ne 1 ]; then
    echo "run-claude.sh: --del only removes the isolated config; not valid with -H or -c" >&2
    exit 2
  fi
  PROJ="$(basename "$WORK_DIR")"
  DEL_DIR="$HOME/.docker-agent/$PROJ/claude"
  if [ ! -e "$DEL_DIR" ]; then
    echo "--del: nothing to delete at $DEL_DIR"; exit 0
  fi
  printf 'About to delete %s\nType the project name (%s) to confirm: ' "$DEL_DIR" "$PROJ"
  read -r _ans
  [ "$_ans" = "$PROJ" ] || { echo "aborted"; exit 1; }
  rm -rf "$DEL_DIR"
  [ -e "$DEL_DIR.json" ] && rm -f "$DEL_DIR.json"        # claude's sibling .json
  rmdir "$HOME/.docker-agent/$PROJ" 2>/dev/null || true  # prune parent if now empty
  echo "deleted $DEL_DIR"
  exit 0
fi

if [ "$ISOLATE" -eq 1 ]; then
```

## Task 6: End-to-end runtime verification

**Files:** none (verification only)

> Uses a fake project work dir + fake `~/.docker-agent` entries. The runner derives the
> config path from `basename "$WORK_DIR"`, so a temp work dir named e.g. `zztest` maps to
> `~/.docker-agent/zztest/<agent>`. `--del` exits before any container launch, so no
> Docker is needed. Confirmation is piped to the runner's `read`.

- [ ] **Step 1: Set up a fake entry and confirm match deletes it**

```bash
proj="zzdeltest-$$"
work="$(mktemp -d)/$proj"; mkdir -p "$work"
mkdir -p "$HOME/.docker-agent/$proj/claude"
echo dummy > "$HOME/.docker-agent/$proj/claude.json"
echo "$proj" | ./run-claude.sh -w "$work" --del
echo "exists after: $( [ -e "$HOME/.docker-agent/$proj" ] && echo yes || echo no )"
```
Expected: prints `deleted .../$proj/claude`; `exists after: no` (dir + sibling json
gone, empty parent pruned).

- [ ] **Step 2: Mismatch aborts, nothing deleted**

```bash
proj="zzdeltest2-$$"
work="$(mktemp -d)/$proj"; mkdir -p "$work"
mkdir -p "$HOME/.docker-agent/$proj/claude"
echo "WRONG" | ./run-claude.sh -w "$work" --del; echo "rc=$?"
echo "exists after: $( [ -d "$HOME/.docker-agent/$proj/claude" ] && echo yes || echo no )"
rm -rf "$HOME/.docker-agent/$proj"
```
Expected: prints `aborted`, `rc=1`, `exists after: yes` (untouched).

- [ ] **Step 3: Sibling agent + parent survive when another agent remains**


remains).
